# T-Guardian: AI 기반 상표 침해 모니터링 시스템
## 통합 프로젝트 문서 (v3.0 Final)

**작성일:** 2025.12.06  
**프로젝트명:** T-Guardian (Trademark Guardian)  
**목적:** AI를 활용한 오프라인·온라인 상표 침해 자동 탐지 및 변리사 보조 시스템

# I. 프로젝트 개요

## 1.1. 배경 및 Pain Point

### 현실의 문제
- **디지털 전환 시대의 상표 침해 증가**: E-commerce, SNS, 메타버스 등 온라인 채널 확대로 상표 침해 사례가 폭발적으로 증가
- **등록 후 관리(Post-registration Management) 공백**: 특허청(KIPRIS)은 상표 등록만 지원하며, 등록 후 침해 모니터링은 권리자 책임
- **수동 모니터링의 한계**: 변리사가 네이버 지도, 쿠팡, 인스타그램 등을 일일이 검색하는 것은 비현실적
- **비정형 데이터 처리 어려움**: 간판 사진, 소셜미디어 이미지, 온라인 상품 사진 등은 텍스트 검색으로 포착 불가능

### 문제의 핵심
기존 시스템은 **"등록"**에만 집중하고, **"침해 감시"**는 사각지대입니다. 특히 오프라인(간판, 매장) 침해는 아예 모니터링 수단이 없습니다.

## 1.2. 핵심 가치 제안 (Value Proposition)

T-Guardian은 다음과 같은 가치를 제공합니다:

1. **24/7 자동 모니터링**: AI가 네이버 지도, 카카오맵, 쿠팡, 인스타그램 등을 24시간 크롤링하여 침해 의심 사례 자동 발견
2. **3가지 유사도 종합 분석**: 한국 상표법에 따라 외관·호칭·관념 유사도를 AI로 정량화 (0~100% 점수)
3. **변리사 보조 도구 (DSS)**: 최종 판단은 변리사가 하되, AI가 침해 후보를 자동으로 선별하여 검토 시간 90% 단축
4. **로컬 데이터 특화**: 한글 상표, 한국 지도 서비스, 한국 법 판례에 최적화 (글로벌 경쟁사와 차별화)
5. **피드백 루프**: 변리사의 판정을 학습하여 모델 정확도 지속 향상

## 1.3. 목표 고객 및 시장

### Primary User (B2B SaaS)
- **대기업 IP팀**: 삼성, LG, 스타벅스 등 브랜드 보호가 중요한 기업
- **변리사 사무소**: 고객사의 상표 침해 모니터링 대행 서비스 제공
- **프랜차이즈 본사**: 가맹점의 무단 브랜드 사용 모니터링

### Secondary User
- **특허청**: 침해 사례 빅데이터 분석 및 정책 수립
- **중소 브랜드**: 저렴한 비용으로 자사 상표 보호

### 시장 규모
- **글로벌 브랜드 보호 시장**: 연 2조 원 규모 (Corsearch, Red Points 등)
- **한국 시장**: 추정 500억 원 (미개척 블루오션)
- **경쟁사 공백**: 오프라인(지도) 침해 탐지는 전 세계적으로도 선례 없음 (First-mover)

# II. 시스템 아키텍처

## 2.1. 전체 시스템 구조

```
┌─────────────────────────────────────────────────────────┐
│               [INPUT SOURCES] 데이터 수집                │
├─────────────────────────────────────────────────────────┤
│  ├─ 공공 데이터: KIPRIS, 지자체 인허가 DB              │
│  ├─ 오프라인: 네이버 지도, 카카오맵 (크롤링)           │
│  ├─ 온라인: 쿠팡, 네이버쇼핑 (크롤링)                   │
│  └─ SNS: 인스타그램 위치 태그, 사용자 제보             │
└─────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────┐
│          [PHASE 1] 데이터 수집 & 전처리                 │
├─────────────────────────────────────────────────────────┤
│  1. 크롤링 시스템 (Selenium/Playwright)                │
│  2. 이미지 저장소 (PostgreSQL + S3)                    │
│  3. 메타데이터 추출 (위치, 카테고리, 상표명)            │
│  4. 중복 제거 (Perceptual Hashing)                     │
└─────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────┐
│       [PHASE 2] 업종 필터링 (Nice 분류 매핑)            │
├─────────────────────────────────────────────────────────┤
│  - KIPRIS 등록 상표의 Nice 분류 확인                   │
│  - 크롤링 데이터의 업종(KSIC)과 매칭                   │
│  - 관련 없는 업종 제거 → AI 분석 비용 90% 절감         │
└─────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────┐
│        [PHASE 3] AI 다차원 분석 (3-Track Model)         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─ [Track 1] 호칭 모델 (Phonetic Similarity) ───┐    │
│  │  - Input: OCR 추출 한글 텍스트                  │   │
│  │  - KoG2P (음운 변환) + Siamese LSTM            │   │
│  │  - Levenshtein Distance 계산                   │   │
│  │  - Output: 호칭 유사도 0~100%                  │   │
│  └──────────────────────────────────────────────┘    │
│                                                         │
│  ┌─ [Track 2] 외관 모델 (Visual Similarity) ────┐    │
│  │  - Input: 이미지 (간판, 로고, 상품 사진)       │   │
│  │  - YOLOv8 (로고 객체 탐지)                     │   │
│  │  - Siamese Network + Triplet Loss (유사도)    │   │
│  │  - Output: 외관 유사도 0~100%                  │   │
│  └──────────────────────────────────────────────┘    │
│                                                         │
│  ┌─ [Track 3] 관념 모델 (Conceptual Similarity) ─┐   │
│  │  - Input: 상표명 텍스트                         │   │
│  │  - OpenAI/Solar Embedding API                  │   │
│  │  - Cosine Similarity 계산                      │   │
│  │  - Output: 관념 유사도 0~100%                  │   │
│  └──────────────────────────────────────────────┘    │
│                                                         │
└─────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────┐
│         [DECISION ENGINE] 침해 위험도 판정              │
├─────────────────────────────────────────────────────────┤
│  - 논리 규칙:                                           │
│    IF (외관 > 80% AND 호칭 > 80%) → 🔴 HIGH           │
│    IF (외관 > 75% OR 호칭 > 75%) AND 관념 > 70%       │
│       → 🟡 MEDIUM                                       │
│    ELSE → 🟢 LOW                                        │
│  - 출력: Risk Level + 3가지 점수                       │
└─────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────┐
│        [RAG 기반 리포트 생성] LLM Report Engine          │
├─────────────────────────────────────────────────────────┤
│  1. Vector DB에서 관련 판례 검색 (Hybrid Retrieval)    │
│  2. LLM에 JSON 데이터만 제공 (환각 방지)                │
│  3. 구조화된 리포트 자동 생성                           │
│  4. 판례 인용 강제 (Citation System)                   │
└─────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────┐
│        [변리사 대시보드] 최종 검수 & 조치               │
├─────────────────────────────────────────────────────────┤
│  - 침해 의심 상표 리스트 (위험도순 정렬)               │
│  - 각 건의 상세 분석 리포트 (이미지 + 점수 + 판례)     │
│  - 변리사 최종 판정 (침해/비침해/보류)                  │
│  - 법적 조치 제안 (경고장, 소송, 특허청 신청)          │
│  - 피드백 저장 → 모델 재학습                           │
└─────────────────────────────────────────────────────────┘
```

## 2.2. 3-Stage Funnel: 효율성의 핵심

T-Guardian의 가장 큰 강점은 **"3단계 깔때기"** 방식입니다.

### 왜 3단계인가?
만약 모든 이미지를 AI로 분석하면 비용과 시간이 폭발합니다. 예를 들어 서울 강남구에만 10만 개 업체가 있다면, 전부 분석하는 것은 비현실적입니다.

### 해결책: 점진적 필터링
```
[전체 데이터: 10만 건]
   ↓ Phase 1: 공공데이터로 베이스 구축
[필터링 후: 5만 건] (폐업 제외)
   ↓ Phase 2: Nice 분류로 업종 필터링
[필터링 후: 5천 건] (관련 업종만)
   ↓ Phase 3: AI 분석 (비용 집중)
[최종 의심 사례: 500건] (변리사 검토)
```

이렇게 하면 AI 분석 비용을 **95% 절감**하면서도 정확도는 유지합니다.

# III. 데이터 수집 전략

## 3.1. Layer 1: 공공 데이터 (Foundation)

### 3.1.1. KIPRIS API
- **출처**: 특허청 공식 API
- **데이터**: 전체 등록 상표 (1,000만 건 이상)
- **활용**: 등록 상표를 "Ground Truth"로 활용
- **갱신 주기**: 매주 1회
- **포함 정보**:
  - 상표명 (한글, 영문, 도형)
  - Nice 분류 (업종)
  - 출원일, 등록일
  - 지정 상품/서비스

### 3.1.2. 지자체 인허가 데이터
- **출처**: data.go.kr (공공데이터포털)
- **데이터**: 전국 사업자 등록 정보
- **활용**: 신규 개업한 업체 실시간 파악
- **포함 정보**:
  - 상호명
  - 업종 (KSIC 코드)
  - 주소, 좌표
  - 개업일

### 3.1.3. Nice 분류 매핑
KSIC (한국표준산업분류)와 Nice 분류를 매핑하여 업종 필터링:

| KSIC 코드 | 업종명 | Nice 분류 | 예시 |
|----------|--------|----------|------|
| I56111 | 일반 음식점업 | Nice 43 (식음료 서비스) | 스타벅스 |
| C10 | 식품 제조업 | Nice 30 (가공 식품) | 오리온 |
| C13 | 의복 제조업 | Nice 25 (의류) | 나이키 |

이를 통해 **관련 없는 업종은 AI 분석 대상에서 제외** → 비용 90% 절감

## 3.2. Layer 2: 오프라인 데이터 (지도 서비스)

### 3.2.1. 네이버 지도 (Naver Map)
**타겟**: 전국 매장 간판, 로드뷰 이미지

#### 크롤링 방법
- **Tool**: Selenium (Headless Chrome)
- **전략**: 
  1. 지역별 Grid 검색 (300m 단위)
  2. Quad-Tree 알고리즘으로 효율적 탐색
  3. 로드뷰 자동 캡처
- **속도**: 시간당 100~200장 (MVP 기준)
- **법적 이슈**: robots.txt 준수, API 대안 검토 중

#### 수집 데이터
- 매장 외관 사진 (로드뷰)
- 간판 클로즈업 사진
- 메타데이터: 위치 좌표, 상호명, 카테고리

### 3.2.2. 카카오맵
- 네이버 지도와 동일한 방식
- 네이버 차단 시 백업 소스

### 3.2.3. 크롤링 대안 전략 (중요!)

**문제**: 네이버/카카오가 크롤링을 차단할 경우 시스템이 멈출 수 있음

**해결책**: 다층 방어 전략

#### 대안 1: 인스타그램 위치 태그
- **원리**: 사람들이 가게에서 사진을 찍고 위치 태그 남김
- **가치**: 네이버보다 더 최신의, 날것의 증거 사진 (메뉴판, 냅킨, 인테리어)
- **접근**: 해시태그 (#맛집, #핫플) + 지역명 조합

#### 대안 2: 카드사 가맹점 데이터 (강력 추천)
- **출처**: 삼성카드, 신한카드, SKT Geovision
- **데이터**: 가맹점명, 업종, 주소, 매출 규모
- **장점**:
  - ✅ **합법적** (정당하게 구매)
  - ✅ **정확함** (실제 영업 중인 가게만)
  - ✅ **폐업 필터링** (매출 발생 기준)
- **비용**: 초기 무료 → 시리즈 A 이후 구매

#### 대안 3: 생활불편신고 & 안전신문고
- **데이터**: 신규 영업 신고 리스트
- **활용**: 새로 생긴 가게만 타겟팅 → 트래픽 1/1000 감소

### 3.2.4. 수집 데이터 품질 관리
- **중복 제거**: Perceptual Hashing (pHash)
- **품질 필터링**: 저해상도 이미지 제외
- **메타데이터 검증**: GPS 좌표, 상호명 일치 여부

## 3.3. Layer 3: 온라인 마켓플레이스

### 3.3.1. 쿠팡
- **타겟**: 상품명, 상품 이미지, 판매자 정보
- **전략**: 카테고리별 크롤링, OCR로 패키지 텍스트 추출

### 3.3.2. 네이버 쇼핑
- **타겟**: 스마트스토어 상품
- **전략**: 키워드 검색 + 변형 검색 (예: "나이키" → "나익", "Niku")

### 3.3.3. 알리익스프레스, 쿠팡로켓
- **글로벌 짝퉁 유통**: 중국발 위조품 주요 경로
- **전략**: 이미지 위주 수집 (OCR 신뢰도 낮음)

## 3.4. Layer 4: SNS

### 3.4.1. 인스타그램
- **전략**: 위치 태그, 해시태그 검색
- **가치**: 사용자 제보 성격 (실시간성 높음)

### 3.4.2. 페이스북 마켓플레이스
- **타겟**: C2C 거래 (개인 간 거래)
- **특징**: 불법 복제품 유통 많음

# IV. AI 모델 설계

## 4.1. Track 1: 호칭 모델 (Phonetic Similarity)

### 목표
한국 상표법의 "호칭 유사성" 판단: **청각적으로 혼동 가능한가?**

### 처리 과정
```
[Input] 텍스트 A: "스타벅스", 텍스트 B: "스타박스"
   ↓
[1] OCR 추출 (EasyOCR / Naver Clova)
   ↓
[2] 한글 자모 분해 (jamotools)
   - "스타벅스" → ['ㅅ', 'ㅡ', 'ㅌ', 'ㅏ', 'ㅂ', 'ㅓ', 'ㄱ', 'ㅅ', 'ㅡ']
   ↓
[3] G2P 음운 변환 (KoG2P)
   - 음운 규칙 적용:
     * 비음화: "국물" → [궁물]
     * 구개음화: "같이" → [가치]
     * 연음: "먹어" → [머거]
   ↓
[4] Siamese LSTM
   - 두 시퀀스를 각각 LSTM에 입력
   - Feature Vector 추출 (128-dim)
   - Manhattan Distance 계산
   ↓
[5] Output: 호칭 유사도 82%
```

### 모델 아키텍처
- **Base**: Siamese Network + LSTM/GRU
- **Loss**: Contrastive Loss 또는 Triplet Loss
- **학습 데이터**:
  - Positive Pair: 실제 침해 사례 ("스타벅스" ↔ "스타박스")
  - Negative Pair: 무관한 상표 ("스타벅스" ↔ "맥도날드")

### 정확도 목표
- **MVP**: 90~95%
- **검증 방법**: 특허청 거절 이유 통지서 데이터와 비교

## 4.2. Track 2: 외관 모델 (Visual Similarity)

### 목표
한국 상표법의 "외관 유사성" 판단: **시각적으로 혼동 가능한가?**

### 처리 과정
```
[Input] 이미지 (간판 사진)
   ↓
[1] 로고 객체 탐지 (YOLOv8)
   - Bounding Box 추출
   - 로고 영역만 Crop
   ↓
[2] 전처리
   - 크기 정규화 (224x224)
   - 색상 보정
   ↓
[3] Siamese Network (Triplet Loss)
   - Anchor: 등록 상표 이미지
   - Positive: 동일 브랜드 변형
   - Negative: 다른 브랜드
   ↓
[4] Feature Embedding (ResNet50 백본)
   - 512-dim Vector 추출
   ↓
[5] Cosine Similarity 계산
   ↓
[6] Output: 외관 유사도 45%
```

### OCR + 외관 복합 분석
간판에는 "로고 + 텍스트"가 함께 있으므로:
1. **Text Detection**: CRAFT/EasyOCR로 텍스트 영역 탐지 → Track 1, 3으로 전달
2. **Logo Detection**: YOLOv8로 로고 영역 탐지 → Track 2에서 처리
3. **Weighted Fusion**: 최종 점수를 가중 평균

### 학습 데이터 전략
- **Phase 1**: LogoDet-3K (15만 장, 글로벌) - 기초 학습
- **Phase 2**: TMID (1.7만 쌍, 알리페이) - Metric Learning
- **Phase 3**: AI Hub 유사 상표 데이터 (27만 건, 한국) - Fine-tuning
- **Phase 4**: 합성 데이터 (Style Transfer) - Data Augmentation

## 4.3. Track 3: 관념 모델 (Conceptual Similarity)

### 목표
한국 상표법의 "관념 유사성" 판단: **의미상으로 혼동 가능한가?**

### 처리 과정
```
[Input] 텍스트 A: "Apple", 텍스트 B: "사과"
   ↓
[1] LLM Embedding API (OpenAI / Solar)
   - "Apple" → [0.01, 0.95, -0.34, ...] (1536-dim)
   - "사과" → [0.02, 0.93, -0.31, ...]
   ↓
[2] Cosine Similarity 계산
   ↓
[3] Output: 관념 유사도 98%
```

### 특수 케이스 처리
- **번역/음차**: "LION" ↔ "라이온" → 높은 유사도
- **의미 확장**: "Royal" ↔ "King", "Emperor" → 중간 유사도
- **동음이의어**: "Apple(사과)" vs "Apple(회사)" → 컨텍스트 분석

### Nice 분류 보조
동일한 단어라도 Nice 분류가 다르면 침해 아님:
- "Apple" (Nice 9: IT) vs "Apple" (Nice 31: 과일) → 비침해

→ **BERT Classifier로 Nice 분류 자동 예측** (45개 분류)

## 4.4. Decision Engine: 침해 위험도 판정

### 통합 로직
```python
# Weighted Max-Pooling 방식
final_score = max(w_p * phonetic_score, 
                  w_v * visual_score, 
                  w_c * conceptual_score)

# 또는 Weighted Average (선택)
final_score = (w_p * phonetic + w_v * visual + w_c * conceptual) / 3

# 가중치 (도메인 전문가 협의)
w_p = 0.4  # 호칭이 가장 중요
w_v = 0.35
w_c = 0.25
```

### Risk Level 분류
```python
if final_score >= 80:
    risk_level = "HIGH" (🔴)
    action = "변리사 즉시 검토 필요"
elif final_score >= 50:
    risk_level = "MEDIUM" (🟡)
    action = "추가 조사 권장"
else:
    risk_level = "LOW" (🟢)
    action = "모니터링 지속"
```

### 한국 상표법 기준
**대법원 판례**: "어느 하나만 유사해도 전체적으로 오인·혼동 가능하면 유사상표"

→ 따라서 **Max-Pooling** 방식이 법리에 더 부합

## 4.5. RAG 기반 리포트 생성

### 문제: LLM의 환각(Hallucination)
LLM에게 "이 상표가 침해인지 판단해줘"라고 자유롭게 물으면 소설을 씁니다.

### 해결책: 구조화된 프롬프트 + RAG

#### Step 1: JSON 데이터만 제공
```json
{
  "target_trademark": "스타벅스",
  "found_trademark": "스타박스",
  "phonetic_similarity": 85,
  "visual_similarity": 45,
  "conceptual_similarity": 10,
  "risk_level": "HIGH",
  "nice_class": "43 (식음료)"
}
```

#### Step 2: Hybrid Retrieval (판례 검색)
1. **Keyword Search (BM25)**: "호칭 유사", "음운", "오인·혼동"
2. **Semantic Search (Dense Vector)**: 의미적으로 유사한 판례
3. **Reciprocal Rank Fusion (RRF)**: 두 결과 통합

#### Step 3: 프롬프트 엔지니어링
```
[System]
당신은 지식재산권 전문 보조원입니다.
아래 [JSON 데이터]만을 사용하여 문장을 완성하십시오.
절대 JSON에 없는 사실을 추가하거나 유추하지 마십시오.

[참고 판례]
- 대법원 2000후1234: "호칭은 전체적·직관적으로 관찰"
- 대법원 1993후544: "이격적 관찰 필요"

[JSON 데이터]
{ ... }

[출력 형식]
1. 호칭 분석: ...
2. 외관 분석: ...
3. 관념 분석: ...
4. 법적 평가: ...
[⚠️ 변리사 최종 검토 필수]
```

#### Step 4: 자동 생성 리포트 예시
```
1. 호칭 분석
   두 상표는 호칭 유사도가 85%로 매우 높습니다.
   '벅'과 '박'의 모음 차이는 있지만, 청감상 혼동 가능성이 큽니다.
   [참조: 대법원 2000후1234]

2. 외관 분석
   외관 유사도는 45%로 중간 수준입니다.
   로고의 색상은 유사하나, 형태는 다릅니다.

3. 관념 분석
   관념 유사도는 10%로 낮습니다.
   (스타벅스: 커피 브랜드, 스타박스: 불명확)

4. 법적 평가
   호칭 유사도가 80% 이상이므로, 한국 상표법상 침해 가능성이 높습니다.
   다만 외관과 관념은 차이가 있어 추가 검토가 필요합니다.
   [⚠️ 변리사 최종 검토 필수]
```

## 5.3. 데이터 한계 및 해결책

### 문제 1: 한글 상표 데이터 부족
글로벌 데이터셋(LogoDet-3K, TMID)은 영어·중국어 위주

**해결책**:
- AI Hub 데이터를 메인으로 사용
- 한글 특화 OCR (Naver Clova) 활용
- Few-shot Learning으로 소수 데이터 극복

### 문제 2: 법적 기준과 데이터 라벨링 기준 불일치
데이터셋의 "유사" 기준 ≠ 한국 상표법의 "이격적·직관적 관찰"

**해결책**:
- 변리사의 실제 판정 데이터를 Feedback Loop로 수집
- Active Learning: 모델이 헷갈리는 건 변리사에게 물어보기
- RLHF (Human-in-the-loop): 지속적 모델 개선

### 문제 3: 현장 이미지 부족
특허청 데이터는 "깔끔한 원본"뿐. 우리가 필요한 건 "간판 사진, 냅킨 로고"

**해결책 1: 합성 데이터 (Style Transfer)**
```
[깨끗한 원본 로고]
   ↓ (배경 이미지에 합성)
[빈 간판] + [로고] → [간판 사진 10만 장 생성]
   ↓ (Data Augmentation)
[회전, 흐림, 조명 변화 적용]
```

**해결책 2: 구글 이미지 크롤링**
키워드: "짝퉁 로고", "funny brand knockoffs", "유사 상표 사례"

### 8.1.3. MarkMonitor
- **강점**: 엔터프라이즈급, 도메인 모니터링
- **약점**: 매우 비쌈, 중소기업 접근 어려움

## 8.2. 한국 경쟁사

### 8.2.1. KIPRIS (특허청)
- **강점**: 공공 데이터, 신뢰도 높음
- **약점**: 등록만 지원, 침해 모니터링 없음

- **강점**: 국내 변리사 네트워크
- **약점**: 수동 검색, AI 활용 미흡

| 구분 | 글로벌 경쟁사 | KIPRIS | T-Guardian |
|------|--------------|--------|------------|
| **오프라인 데이터** | ❌ | ❌ | ✅ (네이버 지도, 카카오맵) |
| **한글 특화** | ❌ | △ | ✅ (AI Hub, 한글 OCR) |
| **AI 3-Track** | △ | ❌ | ✅ (외관·호칭·관념) |
| **법리 정합성** | △ | ✅ | ✅ (한국 판례 RAG) |
| **변리사 협력** | △ | ❌ | ✅ (보조 도구 포지셔닝) |

3. **로컬 데이터**: 한글 상표, 한국 지도, 한국 OCR 특화

# IX. 결론: 성공의 핵심

## 9.1. MVP 성공 조건

1. **크롤링을 먼저 완성** → 실제 데이터 확보
2. **Vision 모델은 최소한 동작** → 85% 정도면 충분
3. **변리사 피드백 수집** → 다음 단계 방향 결정
4. **반복적 개선** → 한 번에 완벽하게 하지 말 것

## 9.2. 장기 성공 조건

1. **변리사 신뢰 확보**: "AI는 보조, 최종 판단은 변리사"
2. **한국 특화 강화**: AI Hub 데이터, 판례 DB 연동
3. **피드백 루프**: 변리사 판정 → 모델 개선 → 정확도 향상
4. **법적 안전성**: 크롤링 대안 확보, 약관 강화

## 9.3. 최종 비전

**"세상의 모든 상표 침해를 찾는 시스템"**

- AI가 24시간 전국 오프라인·온라인을 모니터링
- 변리사는 AI가 선별한 의심 사례만 검토
- 검토 시간 90% 단축 → 더 많은 고객 확보 가능
- 피드백 루프로 정확도 지속 향상 → Data Flywheel
